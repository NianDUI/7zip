# M1-T9 报告：列表性能 gate（M1 出口闸门）

> 目标：用大条目数压测 `SZFolderCore` 快照模型，对照 `01-architecture.md §5.4` 内存预算判定是否需返工懒加载，建立性能基线。**M1 出口闸门：不达标即回炉数据结构，不带病进 M2。**
> 结论：**通过。快照模型内存/速度达标，不需返工懒加载。**
> 复现：`Mac/SevenZipKit/build_perf_test.sh`。基线机：macOS arm64。

## 方法

造扁平归档（单目录包裹 N 个空文件，模拟单层海量条目的渲染压力）1万/10万/100万；`perf_test.cpp` 打开后下钻到内容层，测：打开延迟（Open+proxy 树+当前层快照+默认排序）、Size/Name 排序耗时、常驻内存（`TASK_VM_INFO.phys_footprint`，可随释放回落）、峰值（`getrusage ru_maxrss`，单调）。

## 结果（实测基线）

| 条目数 | 打开延迟 | Size 排序 | Name 排序 | 常驻(快照) | 常驻均摊 | 峰值 maxrss |
|---|---|---|---|---|---|---|
| 1万   | 15 ms  | 4 ms   | 3 ms   | 5.8 MB   | (固定开销主导) | 11.8 MB |
| 10万  | 54 ms  | 53 ms  | 52 ms  | 47.3 MB  | **424 B/条目** | 52.9 MB |
| 100万 | 570 ms | 412 ms | 406 ms | 329.8 MB | **339 B/条目** | **469.6 MB** |

> 均摊随规模下降（固定开销被摊薄），100 万档 339 B/条目最具代表性。

## §5.4 预算判定

| 指标 | 预算 | 100 万实测 | 判定 |
|---|---|---|---|
| 峰值常驻 | ≤ 600 MB / 100 万 | 469.6 MB | ✅ |
| 每条目均摊（常驻） | ≤ 512 字节 | 339 B | ✅ |
| 失控增长 | 无 | 线性（10万→100万 ≈ ×7，且均摊下降） | ✅ |

**两项预算均达标 → 快照模型可用，不触发懒加载返工。** （懒加载 `02 §5.4` 作为 M5 进一步降延迟的可选优化，非 M1 必需。）

## 过程中的优化（gate 抓出的热点）

1. **排序键预转**：原始实现 `CompareFileNames_ForFolderList` 每次比较都 `UTF-8→UString` 转换，`O(n log n)` 次。改为 `applySort` 内 `O(n)` 预转 `nameKeys`/`extKeys` + 索引排序 → 100 万 Name 排序 **1440 ms → 406 ms（≈3.5×）**。
2. **in-place 重排**：按排序索引原地置换 `items`（cycle-following），省掉一份 100 万 `SZCoreItem` 副本（~80 MB）→ 峰值 **553 MB → 470 MB**。
3. **常驻 vs 峰值区分**：排序瞬时键属临时，`phys_footprint` 实测排序后回落（329→316 MB）。512 B 均摊预算针对常驻（339 B 达标），峰值另以 600 MB 绝对线考核（470 MB 达标）。

## 滚动帧率（60fps）

`NSTableView` view-based 虚拟模式只渲染可见行（约 30 行），与总条目数解耦；`SZPanelController` dataSource 是 `O(1)` 数组访问（`model.items[row]`），不随条目数劣化。故帧率瓶颈不在数据规模。实际 60fps 滚动在桌面图形会话验证（headless 不可测渲染），数据结构侧无阻塞。

## CI 性能回归基线（写入闸门）

后续每次改动应跑 `build_perf_test.sh`，对照本基线（100 万档：打开 ≤ ~700ms、排序 ≤ ~500ms、常驻均摊 ≤ 512B、峰值 ≤ 600MB）；超阈告警。

## 残留（不阻塞 M1，记 M5 性能收尾）

- 打开延迟 570 ms/100 万：主要在 proxy 树构建（Agent）+ 快照 `GetProperty`（×7×N）。懒加载属性可进一步降，但当前达标。
- 排序 406 ms/100 万：预转键已优化；如需更快可缓存排序键（代价常驻内存），当前不必要。
